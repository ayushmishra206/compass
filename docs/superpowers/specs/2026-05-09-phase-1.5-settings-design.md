## 2. Scope

### 2.1 In scope

#### Data + crypto core (`packages/core/src/crypto/`)

- **`getActiveCredentials()` shape detection** — same `chrome.storage.local['llm.creds.v1']` key, dual-shape: raw `LlmCredentials` JSON OR `EncryptedSecret` envelope. Read tries `EncryptedSecretSchema.safeParse` first; on match requires a cached passphrase to decrypt; otherwise falls through to raw `LlmCredentialsSchema`.
- **`setActiveCredentials()` shape-aware writes** — when storage currently holds an envelope, re-encrypts and writes envelope; otherwise writes raw. Existing call sites work transparently in both modes.
- **`enableEncryption(passphrase)`** — reads raw, encrypts to envelope, atomically replaces storage value, caches passphrase in `chrome.storage.session`.
- **`disableEncryption(currentPassphrase)`** — verifies passphrase via decrypt, writes raw, removes session cache.
- **`unlockCredentials(passphrase)`** — verifies passphrase via decrypt, caches passphrase in `chrome.storage.session`.
- **`lockCredentials()`** — removes session cache.
- **`isEncryptionEnabled()` / `isLocked()`** — non-throwing inspectors used by the shell store boot path.
- **`LlmCredentialsLocked` error class** — exported from `@compass/core`. Thrown by `getActiveCredentials` when storage holds an envelope and session has no cached passphrase.
- **`packages/core/src/crypto/passphrase.ts`** — new file with `MIN_PASSPHRASE_LENGTH = 12`, `passphraseStrength(p)` returning `'weak' | 'medium' | 'strong'`, and `passphraseError(p)` returning a single-line error or null.
- **Lint rule preserved** — `no-direct-credentials-storage` continues to scope `chrome.storage.local['llm.creds.v1']` access (and now `chrome.storage.session['llm.creds.v1.kek']` access) to `packages/core/src/crypto/credentials.ts`.

#### Shell Zustand store ([`apps/extension/app/state/shell.ts`](../../../apps/extension/app/state/shell.ts))

New slice fields and actions:

- `encryptionEnabled: boolean` — current `llm.creds.v1` is an envelope.
- `locked: boolean` — encryption is enabled AND no passphrase cached in `chrome.storage.session`.
- `unlockHint: boolean` — transient one-shot signal: when true, ProfileDrawer's unlock form scrolls into view and grabs focus; cleared after acknowledgment.
- `refreshLockState()` — re-reads chrome.storage truth and updates the two booleans. Called once on App mount.
- `unlock(passphrase)` — calls `unlockCredentials(passphrase)`, sets `locked: false, unlockHint: false`.
- `lock()` — calls `lockCredentials()`, sets `locked: true`.
- `requestUnlock()` — opens ProfileDrawer (`drawer = { open: true, kind: 'profile' }`) and sets `unlockHint: true`.
- `setEncryptionState(enabled, locked)` — called by EncryptionSection after opt-in/opt-out succeeds.
- `clearUnlockHint()` — called by the unlock form on focus-acknowledge / drawer close.

**Persistence:** lock-state fields are NOT persisted by the existing Zustand `persist` middleware — they're derived from `chrome.storage.local` / `chrome.storage.session` truth via `refreshLockState()` on boot. Persisting them would cause stale-state bugs across browser restarts.

#### UI components (`apps/extension/app/components/credentials/`)

Three pure-data-flow shared components. None call `setActiveCredentials` directly — persistence is the parent's responsibility.

- **`<KeyValidator>`** — provider chooser + apiKey input + validate button + error display + `onValidated(provider, key)` callback. Used by OnboardingDrawer step 2, ProfileDrawer Add-another, and ProfileDrawer Rotate (with `lockProvider` to collapse the chooser to a label).
- **`<PassphraseSetForm>`** — passphrase input + confirm field + strength meter + `onSet(passphrase)` callback. Used by OnboardingDrawer step 3 and ProfileDrawer Encryption opt-in. Submit disabled until `passphrase.length >= MIN_PASSPHRASE_LENGTH && passphrase === confirm`.
- **`<PassphraseConfirmForm>`** — single passphrase input + `onConfirm(passphrase)` callback. Used by ProfileDrawer Encryption opt-out and ProfileDrawer Unlock form. Surfaces `Wrong passphrase` inline error on rejected `onConfirm`.

All three render inline within the parent's section (no portal/modal). Parent owns open/close lifecycle and persistence.

**`ConnectedProvidersSection`** has two render branches:

1. **Locked branch** (`encryptionEnabled && locked`): renders `<PassphraseConfirmForm submitLabel="Unlock" onConfirm={p => useShell.getState().unlock(p)} />` + a "Forgot passphrase?" link below. On `unlockHint === true`, the form grabs focus + scrolls into view, then calls `clearUnlockHint()`. No row list shown until unlocked.
2. **Unlocked branch:** renders the row list (one row per populated provider) with mask + relative `lastValidatedAt` + default badge + ⋯ menu. The ⋯ menu opens an inline action row beneath the target row with: **Validate now** | **Set as default** (only on non-default rows) | **Rotate** | **Remove**. Single-action-row-open invariant: opening a second auto-collapses the first. Below the rows, a `[ + Add another provider ]` button expands an inline `<KeyValidator>` with `availableProviders = providerIds \ creds`.

Per-row action semantics:

- **Validate now** — calls `rpc('llm.validateKey', { provider, apiKey: row.apiKey })` directly (we have the key); on success writes back via `setActiveCredentials({ ...creds, [provider]: { ...row, lastValidatedAt: nowIso } })`; on failure surfaces inline error.
- **Set as default** — single-action: `setActiveCredentials({ ...creds, default: provider })`.
- **Rotate** — expands inline `<KeyValidator providers={[provider]} initialProvider={provider} lockProvider />`; on `onValidated(provider, newKey)` writes `setActiveCredentials({ ...creds, [provider]: { apiKey: newKey, addedAt: row.addedAt, lastValidatedAt: nowIso } })`. Atomic replace; existing `addedAt` is preserved.
- **Remove** — expands an inline confirm strip. If the target row is the current default AND other providers exist, renders a radio group of remaining providers ("Pick new default") and a single "Remove and set new default" button. If target is default with no other providers, confirms with copy "This will leave you with no providers. You'll see onboarding on next reload." On confirm: writes via `setActiveCredentials({ ...creds, [provider]: undefined, default: newDefault })`. If creds becomes empty, also `chrome.storage.local.remove('profile.byokConfigured')` and sets `onboardingLocked = true`.

**Mask format:** `prefix-…suffix4` — chars up to first non-`[A-Za-z0-9]`, then `…`, then last 4 chars (e.g. `sk-…rA73`). Avoids variable-width "•" rendering quirks.

**Relative time:** new tiny pure helper at `apps/extension/app/lib/relativeTime.ts` (~20 lines, no library). "2 minutes ago", "1 day ago", "3 days ago".

**`EncryptionSection`** has two visual states keyed by `useShell((s) => s.encryptionEnabled)`:

1. **Off:** mono label "Off — keys are stored unencrypted in this browser." + `[ Enable encryption ]` button. Click expands inline `<PassphraseSetForm onSet={p => enableEncryption(p).then(() => setEncryptionState(true, false))} />`.

#### Forgotten-passphrase recovery

A "Forgot passphrase?" link rendered below the unlock form in `ConnectedProvidersSection`'s locked branch. Click expands an inline confirmation strip in place of the unlock form:

> This will permanently erase your saved API keys from this browser.
> You'll see the onboarding screen and need to re-paste your keys.
> There is no recovery — this device cannot decrypt the keys without your passphrase.
>
> [ Cancel ] [ Erase keys and start over ]

`[ Cancel ]` returns to the unlock form. `[ Erase keys and start over ]` runs:

```ts
async function recoverFromForgottenPassphrase() {
  await clearActiveCredentials();
  await chrome.storage.session.remove('llm.creds.v1.kek');
  await chrome.storage.local.remove('profile.byokConfigured');
  useShell.setState({
    encryptionEnabled: false,
    locked: false,
    unlockHint: false,
    drawer: { open: false, kind: null },
    onboardingLocked: true,
  });
}
```

OnboardingDrawer pops up via the existing `onboardingLocked` gate. User lands at step 1 ("Welcome to Compass") with no creds — re-paste flow is the standard 3-step path.

## 3. Architecture

### 3.1 Layer overview

Three layers, each with one responsibility:

1. **Data + crypto core** in `packages/core/src/crypto/` — `credentials.ts` grows shape detection + lock/unlock helpers; `keystore.ts` is already shipped and used as-is; `passphrase.ts` is a tiny new file.
2. **Global state** in `apps/extension/app/state/shell.ts` — extended with a lock-state slice (3 new fields + 6 new actions).
3. **UI surface** split across `apps/extension/app/components/credentials/` (3 reusable widgets) + `apps/extension/app/drawers/profile/` (2 ProfileDrawer sections) + extensions to [ProfileDrawer.tsx](../../../apps/extension/app/drawers/ProfileDrawer.tsx), [OnboardingDrawer.tsx](../../../apps/extension/app/drawers/OnboardingDrawer.tsx), [Topbar.tsx](../../../apps/extension/app/components/Topbar.tsx).

### 3.2 Data flow

#### 3.2.1 Storage shape detection

```ts
// packages/core/src/crypto/credentials.ts (Phase 1.5 settings)
import { LlmCredentialsSchema, type LlmCredentials } from '../types/credentials';
import { EncryptedSecretSchema, encrypt, decrypt } from './keystore';

const STORAGE_KEY = 'llm.creds.v1';
const SESSION_KEK_KEY = 'llm.creds.v1.kek';

export class LlmCredentialsLocked extends Error {
  constructor() {
    super('Credentials are encrypted; call unlockCredentials() first');
    this.name = 'LlmCredentialsLocked';
  }
}

export async function getActiveCredentials(): Promise<LlmCredentials> {
  const raw = await chrome.storage.local.get(STORAGE_KEY);
  const value = raw[STORAGE_KEY];
  if (!value) return { default: null };

  // Try encrypted envelope first
  const env = EncryptedSecretSchema.safeParse(value);
  if (env.success) {
    const passphrase = await getCachedPassphrase();
    if (!passphrase) throw new LlmCredentialsLocked();
    const plaintext = await decrypt(env.data, passphrase);
    return LlmCredentialsSchema.parse(JSON.parse(plaintext));
  }

  // Fall through to raw shape
  const rawParsed = LlmCredentialsSchema.safeParse(value);
  return rawParsed.success ? rawParsed.data : { default: null };
}

export async function setActiveCredentials(creds: LlmCredentials): Promise<void> {
  // Shape-aware writes — preserves the user's encryption choice
  const existing = await chrome.storage.local.get(STORAGE_KEY);
  const env = EncryptedSecretSchema.safeParse(existing[STORAGE_KEY]);
  if (env.success) {
    const passphrase = await getCachedPassphrase();
    if (!passphrase) throw new LlmCredentialsLocked();
    const envelope = await encrypt(JSON.stringify(creds), passphrase);
    await chrome.storage.local.set({ [STORAGE_KEY]: envelope });
  } else {
    await chrome.storage.local.set({ [STORAGE_KEY]: creds });
  }
}

export async function clearActiveCredentials(): Promise<void> {
  await chrome.storage.local.remove(STORAGE_KEY);
}
```

#### 3.2.2 Encryption opt-in / opt-out

```ts
export async function enableEncryption(passphrase: string): Promise<void> {
  const current = await chrome.storage.local.get(STORAGE_KEY);
  const value = current[STORAGE_KEY];
  if (EncryptedSecretSchema.safeParse(value).success) {
    throw new Error('Credentials are already encrypted');
  }
  const creds = LlmCredentialsSchema.safeParse(value).success
    ? (value as LlmCredentials)
    : { default: null };
  const envelope = await encrypt(JSON.stringify(creds), passphrase);
  await chrome.storage.local.set({ [STORAGE_KEY]: envelope });
  await chrome.storage.session.set({ [SESSION_KEK_KEY]: passphrase });
}

export async function disableEncryption(currentPassphrase: string): Promise<void> {
  const current = await chrome.storage.local.get(STORAGE_KEY);
  const env = EncryptedSecretSchema.safeParse(current[STORAGE_KEY]);
  if (!env.success) throw new Error('Credentials are not currently encrypted');
  // Will throw on wrong passphrase (AES-GCM auth-tag mismatch)
  const plaintext = await decrypt(env.data, currentPassphrase);
  const creds = LlmCredentialsSchema.parse(JSON.parse(plaintext));
  await chrome.storage.local.set({ [STORAGE_KEY]: creds });
  await chrome.storage.session.remove(SESSION_KEK_KEY);
}
```

**Atomicity.** Each `chrome.storage.local.set` is atomic per key; readers always see either the prior value or the new value. The opt-in flow does `set(envelope)` then `set(passphrase)` — if the second fails (e.g., browser crash), envelope is in storage but session is empty → user is "locked" until they retype, which is the same recovery path the unlock flow uses.

**Wrong-passphrase detection.** `decrypt()` throws on AES-GCM auth-tag mismatch. `unlockCredentials` and `disableEncryption` rely on this single signal — no extra round-trip.

#### 3.2.3 Unlock / lock

```ts
export async function unlockCredentials(passphrase: string): Promise<void> {
  const current = await chrome.storage.local.get(STORAGE_KEY);
  const env = EncryptedSecretSchema.safeParse(current[STORAGE_KEY]);
  if (!env.success) throw new Error('Credentials are not encrypted');
  // Validate by attempting one decrypt — throws on wrong passphrase
  await decrypt(env.data, passphrase);
  await chrome.storage.session.set({ [SESSION_KEK_KEY]: passphrase });
}

export async function lockCredentials(): Promise<void> {
  await chrome.storage.session.remove(SESSION_KEK_KEY);
}

async function getCachedPassphrase(): Promise<string | null> {
  const r = await chrome.storage.session.get(SESSION_KEK_KEY);
  const v = r[SESSION_KEK_KEY];
  return typeof v === 'string' ? v : null;
}

export async function isEncryptionEnabled(): Promise<boolean> {
  const r = await chrome.storage.local.get(STORAGE_KEY);
  return EncryptedSecretSchema.safeParse(r[STORAGE_KEY]).success;
}

export async function isLocked(): Promise<boolean> {
  const enabled = await isEncryptionEnabled();
  if (!enabled) return false;
  const passphrase = await getCachedPassphrase();
  return !passphrase;
}
```

**Threat model note.** The cached value is the raw passphrase string, not the derived KEK. `chrome.storage.session` is in-memory only, browser-session-scoped, and not persisted to disk. An attacker with code execution in the JS heap has many other attack vectors and is out of scope. The earlier foundation spec sketched a "JWK in session" pattern — caching the passphrase string is simpler (no `crypto.subtle.exportKey/importKey` round-trips on every read), with the same security posture given the threat model.

### 3.3 Shell Zustand store extensions

```ts
// apps/extension/app/state/shell.ts (Phase 1.5 settings additions)

// state
encryptionEnabled: boolean;
locked: boolean;
unlockHint: boolean;

// actions
refreshLockState: () => Promise<void>;
unlock: (passphrase: string) => Promise<void>;
lock: () => Promise<void>;
requestUnlock: () => void;
setEncryptionState: (enabled: boolean, locked: boolean) => void;
clearUnlockHint: () => void;

// concrete impl
encryptionEnabled: false,
locked: false,
unlockHint: false,

refreshLockState: async () => {
  const [enabled, locked] = await Promise.all([isEncryptionEnabled(), isLocked()]);
  set({ encryptionEnabled: enabled, locked });
},

unlock: async (passphrase) => {
  await unlockCredentials(passphrase);
  set({ locked: false, unlockHint: false });
},

lock: async () => {
  await lockCredentials();
  set({ locked: true });
},

requestUnlock: () =>
  set({ drawer: { open: true, kind: 'profile' }, unlockHint: true }),

setEncryptionState: (enabled, locked) =>
  set({ encryptionEnabled: enabled, locked }),

clearUnlockHint: () => set({ unlockHint: false }),
```

**Boot path.** [App.tsx](../../../apps/extension/app/App.tsx) gets one new `useEffect` after the existing onboarding-gate effect:

```ts
useEffect(() => {
  void useShell.getState().refreshLockState();
}, []);
```

**`unlockHint` separation rationale.** `locked` tracks a real state (credentials unreadable). `unlockHint` is a one-shot UI signal that the unlock form should grab focus + scroll into view. Without separation, every `locked` observer would trigger scroll/focus on every render. With it, the unlock form is rendered whenever `locked && encryptionEnabled` (so users can unlock proactively from chip click → drawer open) but only auto-focuses when `unlockHint` is true.

### 3.4 Passphrase rules

```ts
// packages/core/src/crypto/passphrase.ts (NEW)
export const MIN_PASSPHRASE_LENGTH = 12;

export function passphraseStrength(p: string): 'weak' | 'medium' | 'strong' {
  if (p.length < MIN_PASSPHRASE_LENGTH) return 'weak';
  if (p.length < 20) return 'medium';
  return 'strong';
}

export function passphraseError(p: string): string | null {
  if (p.length < MIN_PASSPHRASE_LENGTH) {
    return `At least ${MIN_PASSPHRASE_LENGTH} characters.`;
  }
  return null;
}
```

Length-only with high floor. No diversity rules (modern best practice; NIST SP 800-63B drops composition rules). PBKDF2-SHA256 250k iterations (already shipped in `keystore.ts`) provides KDF stretching even for length-floor passphrases. Strength meter dot in `<PassphraseSetForm>` colored red/amber/green by `passphraseStrength()`. Inline hint: "12+ characters. A short sentence works well."

## 7. Risks & mitigations

| Risk                                                                                        | Mitigation                                                                                                                                                                                                                                               |
| ------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Multi-tab divergence: opt-in/wipe in one tab leaves another tab's shell store stale         | Documented as known limitation; future workstream adds `chrome.storage.onChanged` listening to `refreshLockState()`                                                                                                                                      |
| User opts in then forgets passphrase, loses access to all keys                              | Forgotten-passphrase recovery flow ships in Phase 1.5; documented inline + in PR body                                                                                                                                                                    |
| `<KeyValidator>` extraction touches recently-shipped Phase 1.6 OnboardingDrawer             | Refactor preserves all behavior tested in `apps/extension/tests/e2e/shell-onboarding.spec.ts` (T42); regression caught by existing e2e                                                                                                                   |
| Encryption-at-onboard window: keys are raw at-rest between step-2 success and step-3 submit | Documented in PR body. Window is seconds for users who pick encryption immediately. "Skip for now" users stay raw indefinitely so this isn't a regression. Full mitigation would require staging credentials in memory until step 3 — not worth the cost |
| Passphrase string in `chrome.storage.session` rather than derived KEK                       | Threat model: session storage is in-memory only, browser-session-scoped, not on disk. An attacker with code execution in the JS heap has many other attack vectors. Same security posture as the spec's KEK-as-JWK pattern, with simpler implementation  |
| ProfileDrawer becomes long-scrolling                                                        | Drawer already supports vertical scroll; sections separated by hairlines preserve scannability. The `single-action-row-open` invariant prevents simultaneous expanded forms from compounding the scroll length                                           |
