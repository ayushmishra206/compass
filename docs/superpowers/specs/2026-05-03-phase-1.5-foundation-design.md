# Compass — Phase 1.5: Foundation Continuation

**Status:** Draft for review
**Date:** 2026-05-03
**Phase:** 1.5 (Foundation Continuation) per [PRD §17](../../prd.md#17-implementation-phases-and-acceptance-gates)
**Scope:** Three sequential workstreams — direct LLM providers (OpenAI + Anthropic), `chrome.alarms` scheduler with cross-browser shim, multi-key Settings UX with encrypted-storage opt-in.

---

## 1. Purpose

Phase 1 shipped the OpenRouter-only foundation. Phase 1.5 turns that foundation into production-shape multi-provider support, ships a cross-browser alarm scheduler that Phase 2's Daily Agent can plug into, and surfaces the encrypted-storage opt-in UX (the `packages/core/crypto` envelope was shipped + tested at 100% in Phase 1 but unwired).

Three workstreams executed sequentially on independent branches off `master`:

1. **`phase-1.5-providers`** — Direct OpenAI + Anthropic providers; `LlmProvider` abstraction matures (the (μ) decision in the Phase 1 spec absorbed direct providers into 1.5 because Anthropic is the real test of the abstraction). Failover (PRD §7.2 rule 2) becomes operational once two keys are populated.
2. **`phase-1.5-alarms`** — `chrome.alarms` scheduler with idempotent cross-browser reconcile, port-based offscreen keep-alive, and a hybrid testing strategy (unit + Chromium integration + manual cross-browser checklist).
3. **`phase-1.5-settings-encrypted`** — Connected providers section in Settings becomes real (multi-key list, validate, rotate, set-as-default, remove); encrypted-storage opt-in toggle wires the existing keystore into a passphrase + lazy-unlock UX.

Each workstream merges to master independently. Phase 2 work can begin on master after the providers branch lands.

The Phase 1.5 gate ([PRD §17](../../prd.md#17-implementation-phases-and-acceptance-gates), with wording sharpened per Q18):

- `chrome.alarms` fires at local time; alarm listener runs in all 3 browsers (Chrome / Firefox / Safari). Full alarm → SW → offscreen RPC chain verified in Chrome.
- Direct OpenAI + Anthropic providers callable end-to-end; `gate:wired` matrix green for at least one Phase 1.5 release run.
- Settings shows real connected-providers data; encrypted-storage opt-in flow round-trips (unlock → use → relock → unlock again).

#### 3.1.2 Tool-use synthesis for Anthropic structured output

Anthropic's API has no `response_format: { type: 'json_schema' }` equivalent. The structured-output contract (`req.schema` → `LlmResponse.parsed`) is honored via tool-use synthesis:

```ts
// packages/llm/src/providers/anthropic.ts (sketch)
const TOOL_NAME = '__compass_response';

const tools = req.schema
  ? [
      {
        name: TOOL_NAME,
        description: 'Return your response in this exact structured format.',
        input_schema: zodToJsonSchema(req.schema),
      } satisfies Anthropic.Tool,
    ]
  : undefined;

const resp = await client.messages.create({
  model: req.model,
  system: req.system,
  messages: req.messages,
  max_tokens: req.maxOutputTokens,
  temperature: req.temperature,
  tools,
  tool_choice: req.schema ? { type: 'tool', name: TOOL_NAME } : undefined,
});

const toolUse = resp.content.find(
  (b): b is Anthropic.ToolUseBlock => b.type === 'tool_use' && b.name === TOOL_NAME,
);
const text = resp.content
  .filter((b): b is Anthropic.TextBlock => b.type === 'text')
  .map((b) => b.text)
  .join('');

return {
  parsed: toolUse?.input,
  text,
  usage: {
    promptTok: resp.usage.input_tokens,
    cachedTok: resp.usage.cache_read_input_tokens ?? 0,
    completionTok: resp.usage.output_tokens,
  },
  model: resp.model,
  finishReason: mapFinishReason(resp.stop_reason),
};
```

When `req.schema` is undefined (no structured output requested), tools/tool_choice are omitted; `text` is the joined text-block content; `parsed` is undefined.

#### 3.1.3 Failover semantics

`executeTask()` builds the failover chain at call time:

```ts
function buildChain(creds: LlmCredentials): ProviderId[] {
  const order: ProviderId[] = [];
  const def = creds.default;
  const all: ProviderId[] = ['openrouter', 'openai', 'anthropic'];
  if (def && hasKey(creds, def)) order.push(def);
  for (const p of all) if (p !== def && hasKey(creds, p)) order.push(p);
  return order;
}
```

Try each in order; on a failover-trigger error (`LlmKeyMissing` / `LlmRateLimited` / `LlmUnavailable`), advance to next; on success, write the cost ledger row and return; on all-fail, throw the last error. `LlmKeyInvalid` / `LlmTimeout` / `LlmSchemaError` are non-failover errors — surface them immediately. The `callWithSchema` retry wrapper at [validate.ts](../../../packages/llm/src/validate.ts) continues to handle within-provider schema retries (3 attempts) before bubbling out; first 5xx after retries exhaust triggers failover.

### 3.2 Workstream A — Alarms scheduler

#### 3.2.1 Idempotent reconcile

```ts
// packages/integrations/src/scheduling/scheduler.ts (sketch)
export interface DesiredAlarm {
  name: string;
  when: number;
}

export async function ensureAlarms(): Promise<void> {
  const existing = await browser.alarms.getAll();
  const desired = computeDesired();
  for (const d of desired) {
    const e = existing.find((a) => a.name === d.name);
    if (!e || Math.abs(e.scheduledTime - d.when) > 60_000) {
      await browser.alarms.clear(d.name);
      await browser.alarms.create(d.name, { when: d.when });
    }
  }
  for (const e of existing) {
    if (!desired.find((d) => d.name === e.name)) {
      await browser.alarms.clear(e.name);
    }
  }
}
```

`computeDesired()` reads `briefingHour` and `reflectionHour` from `defaults.ts` and computes the next-occurrence epoch ms in local time. The 60-second tolerance avoids gratuitous churn on hour-aligned reschedules.

#### 3.2.2 Cross-browser bootstrap hooks

```ts
// apps/extension/entrypoints/background.ts (Phase 1.5 additions)
import { ensureAlarms, registerAlarmHandlers } from '@compass/integrations/scheduling';

export default defineBackground(() => {

  registerAlarmHandlers(); // wires browser.alarms.onAlarm listener
  void ensureAlarms(); // top-level: runs on every SW wake / FF page reload / Safari SW wake

  browser.runtime.onInstalled.addListener(() => {
    void ensureAlarms();
  });
  if (browser.runtime.onStartup) {
    browser.runtime.onStartup.addListener(() => {
      void ensureAlarms();
    });
  }

```

The same code runs on all three browsers. The reconcile logic doesn't care whether alarms persisted — if they did, `existing` matches `desired`, no-op. If they didn't, `existing` is empty, all desired alarms get created.

### 3.3 Workstream C — Settings + encrypted-storage

#### 3.3.1 Storage shape detection

```ts
// packages/core/src/crypto/credentials.ts (Phase 1.5)
export class LlmCredentialsLocked extends Error {
  constructor() {
    super('Credentials are encrypted; call unlockCredentials() first');
    this.name = 'LlmCredentialsLocked';
  }
}

export async function getActiveCredentials(): Promise<LlmCredentials> {
  const raw = await chrome.storage.local.get(STORAGE_KEY);
  const value = raw[STORAGE_KEY];
  if (!value) return { default: null };

  // Try encrypted envelope first (discriminator: { v: 1, algo: 'AES-GCM-256' })
  const envParsed = EncryptedSecretSchema.safeParse(value);
  if (envParsed.success) {
    const kek = await getCachedKek();
    if (!kek) throw new LlmCredentialsLocked();
    const plaintext = await decryptWithKey(envParsed.data, kek);
    return LlmCredentialsSchema.parse(JSON.parse(plaintext));
  }

  // Raw shape
  const rawParsed = LlmCredentialsSchema.safeParse(value);
  return rawParsed.success ? rawParsed.data : { default: null };
}
```

Phase 1 users with raw `llm.creds.v1` continue to read uninterrupted — `EncryptedSecretSchema.safeParse` returns `success: false` on the raw shape, and the read falls through to `LlmCredentialsSchema.safeParse`. Migration to encrypted is opt-in via Settings; the opt-in flow encrypts the existing raw value and writes the envelope back to the same key.

#### 3.3.2 KEK lifecycle

```ts
const SESSION_KEY = 'llm.creds.v1.kek';

export async function unlockCredentials(passphrase: string): Promise<void> {
  const raw = await chrome.storage.local.get(STORAGE_KEY);
  const env = EncryptedSecretSchema.safeParse(raw[STORAGE_KEY]);
  if (!env.success) throw new Error('No encrypted credentials to unlock');
  const kek = await deriveKey(passphrase, b64decode(env.data.salt));
  // Validate by attempting one decrypt; throws on bad passphrase.
  await crypto.subtle.decrypt(
    { name: 'AES-GCM', iv: b64decode(env.data.iv) as BufferSource },
    kek,
    b64decode(env.data.ct) as BufferSource,
  );
  const jwk = await crypto.subtle.exportKey('jwk', kek);
  await chrome.storage.session.set({ [SESSION_KEY]: jwk });
}

export async function lockCredentials(): Promise<void> {
  await chrome.storage.session.remove(SESSION_KEY);
}

async function getCachedKek(): Promise<CryptoKey | null> {
  const r = await chrome.storage.session.get(SESSION_KEY);
  const jwk = r[SESSION_KEY];
  if (!jwk) return null;
  return crypto.subtle.importKey('jwk', jwk, { name: 'AES-GCM', length: 256 }, false, ['decrypt']);
}
```

`chrome.storage.session` is in-memory only and clears on browser quit, satisfying PRD §5.3's "for the browser session." No idle-timeout auto-lock in Phase 1.5; user can lock manually from Settings.

#### 3.3.3 Settings UX shape

#### 3.3.4 Lazy unlock UX

```ts
// apps/extension/app/hooks/useUnlockState.ts (sketch)
export function useUnlockState() {
  const [locked, setLocked] = useState(false);
  const [encryptionEnabled, setEncryptionEnabled] = useState(false);

  useEffect(() => {
    void checkLockState();
  }, []);

  async function checkLockState() {
    const raw = await chrome.storage.local.get(STORAGE_KEY);
    const isEnc = EncryptedSecretSchema.safeParse(raw[STORAGE_KEY]).success;
    setEncryptionEnabled(isEnc);
    if (isEnc) {
      const session = await chrome.storage.session.get(SESSION_KEY);
      setLocked(!session[SESSION_KEY]);
    } else {
      setLocked(false);
    }
  }

  async function unlock(passphrase: string) {
    await unlockCredentials(passphrase);
    setLocked(false);
  }

  async function lock() {
    await lockCredentials();
    setLocked(true);
  }

  return { locked, encryptionEnabled, unlock, lock };
}
```

The topbar shows a small lock-icon Badge when `locked && encryptionEnabled`. Clicking it opens `UnlockModal`. Credential-needing actions (e.g., the Validate button in Settings) catch `LlmCredentialsLocked` and open `UnlockModal` with a retry callback.

## 8. Risks & mitigations

| Risk                                                                                 | Mitigation                                                                                                                                                                                                                                                       |
| ------------------------------------------------------------------------------------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Anthropic SDK unstable in browser/offscreen context                                  | Implementer subagent verifies `dangerouslyAllowBrowser`-equivalent during Task 5 of providers plan; falls back to direct `fetch` POST if SDK refuses (mirrors a workaround the OpenAI SDK needed pre-`dangerouslyAllowBrowser`).                                 |
| Tool-use synthesis returns no `tool_use` block (model didn't comply)                 | `callWithSchema` wrapper catches missing `parsed`, retries with explicit format-feedback message; on exhaust throws `LlmSchemaError`. Same path as OpenRouter schema failures.                                                                                   |
| Failover loop on 5xx storms                                                          | Failover order is finite (max 3 providers); each attempt is one HTTP call (no inner retry beyond `callWithSchema`'s 3 attempts); total worst-case latency = 3 × (max_attempt × per_call_timeout). Document in spec; add monitoring once Phase 4 telemetry ships. |
| `chrome.alarms` test flakiness in CI                                                 | `gate:alarms` asserts only that alarms are registered with valid `when` (no time-warp); cross-browser persistence asserted via manual checklist. Avoids the entire class of timing-dependent CI flakes.                                                          |
| Port-based keep-alive doesn't survive in Safari MV3 offscreen                        | Safari verification is "alarm fires + listener runs" only (Q18); full chain not required in Phase 1.5. Cross-browser sprint addresses Safari offscreen gaps.                                                                                                     |
| User opts in to encryption then forgets passphrase, loses access to all keys         | Forgotten-passphrase flow clears credentials, reroutes to onboarding. Re-paste is cheap (1-3 keys). UX explicit about irreversibility (Q16 modal copy).                                                                                                          |
| KEK leakage from `chrome.storage.session`                                            | Session storage is in-memory only, not on disk. An attacker with code execution in the offscreen JS heap has many other attack vectors. Threat model accepts this trade-off (PRD §5.3).                                                                          |
| Routing table model strings (`gpt-4o-mini`, `claude-haiku-4-5`) drift before Phase 2 | Trivial config-file edit; not a code change. Bias toward keeping current model strings until a Phase 2+ feature actually fails.                                                                                                                                  |
| Phase 1.5 ships, encrypted-storage UX deferred again                                 | Encrypted-storage is the _third_ workstream and its plan exists from this spec; even if it slips, providers + alarms stand alone. Spec naming convention names it for visibility.                                                                                |
